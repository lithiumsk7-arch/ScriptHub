# Script Hub 統一腳本呼叫介面 規格書

## 1. 專案概述
- **目的**：提供一個具備多頁籤功能的統一 UI 介面，解決多個專案中 `.py`、`.exe`、`.bat` 等腳本檔案分散各處、難以尋找與管理的問題。
- **目標作業系統**：Windows。

## 2. 核心功能需求

### 2.1 多頁籤管理 (Tab Management)
- **多頁籤模式**：使用者可建立多個頁籤來分類與管理腳本。
- **頁籤操作**：
  - **新增**：可建立新的空白頁籤。
  - **修改/重命名**：支援滑鼠雙擊 (Double Click) 頁籤標籤，或透過點擊右鍵選單進行重新命名。
  - **複製**：可完整複製現有頁籤（包含其內部的所有腳本與參數設定）。
  - **刪除**：移除指定的頁籤（亦可透過右鍵選單呼叫）。
  - **排序**：支援以滑鼠左鍵按住拖拉 (Drag & Drop) 的方式調整頁籤的顯示順序。

### 2.2 腳本管理 (Script Management)
- **腳本列表**：每個頁籤內可管理多個腳本，列表介面上需顯示該腳本的「名稱」。
- **新增/修改腳本**：
  - 支援使用「檔案選擇器 (File Picker)」指定本機檔案位置。
  - 支援直接「文字輸入/編輯」檔案路徑。
- **腳本操作**：使用者可隨意新增、刪除、編輯清單內的腳本。
- **排序**：支援以拖拉 (Drag & Drop) 的方式調整腳本在該頁籤內的排列順序。

### 2.3 參數編輯與處理 (Parameter Handling)
- **參數輸入區塊**：點擊腳本後，可編輯其參數。每個腳本配備 **3 個獨立的文字輸入框** 以及 **1 個 `...` (進階編輯) 按鈕**。
- **參數輸入邏輯**：
  - 文字輸入框：允許輸入包含空白的字串。
  - `...` 進階編輯視窗：點擊後彈出獨立的文字編輯視窗。為防呆設計，不論是點擊儲存按鈕，或是直接點擊右上角 `X` 關閉視窗，皆會自動把輸入內容回傳並觸發保存。
- **參數合併**：執行時，這 3 個文字輸入框的內容會與 `...` 編輯視窗內的參數「合併」，作為該腳本的最終輸入。
- **路徑與相容性處理**：
  - **路徑相容**：能自動處理或相容 Windows 反斜線 (`\`) 與 Linux 正斜線 (`/`) 的路徑格式問題。
  - **引號解析與精準傳遞**：能妥善處理被雙引號 (`"`) 或單引號 (`'`) 包圍的參數。**必須確保參數的引號或空字串能精準傳遞給目標腳本**。
    - **執行範例要求**：若參數合併後為 `A.py --par0 66 --par1 "00"`，底層實作 (透過 `shlex.split` 或其他解析器) 必須保證傳遞給 CMD 的參數中，`--par1` 之後接的正是被視為獨立參數的 `"00"`（或等價的字串處理），確保目標腳本拿到的不會是空值或被剝奪引號而導致型別解析錯誤。

### 2.4 執行與回饋 (Execution & Feedback)
- **執行腳本**：點選腳本對應的「執行」按鍵，程式將直接呼叫該腳本，並帶入預先編輯好的合併參數。
- **指令確認**：每次按下執行鈕後，UI 介面必須顯示 **最終傳遞給 Shell 的完整指令 (Raw Command)**，以供使用者確認實際的執行狀態與除錯。

### 2.5 狀態保存 (State Persistence)
- **手動儲存**：介面需提供一個「儲存設定」的按鈕，點擊後立即寫入當下狀態。
- **自動儲存**：每次使用者「關閉此程式」時，或是在任何文字欄位產生「輸入變更」時，系統必須自動且原封不動地保存目前所有的狀態。
- **儲存位置**：儲存的設定檔 (`config.json`) 必須嚴格綁定於「與此程式主程式 (main.py 或 exe) 完全相同的絕對路徑目錄中」，無論使用者在何種路徑下執行指令，都能準確寫入同目錄，維持綠色軟體的特性。

### 2.6 錯誤處理與進階參數防呆 (Error Handling & Parameter Validation)
- **拒絕 Shell Injection (命令注入) 實作**：
  - **技術明訂**：程式底層呼叫必須嚴格使用 `subprocess.Popen(..., shell=False)`。將合併後的參數字串透過 `shlex.split()` 轉換為 List 陣列格式再傳遞，以完全阻絕 `&`, `|`, `>`, `<` 等特殊字元被 OS 誤認為 Shell 語法而引發惡意執行。
- **參數解析驗證**：
  - **技術明訂**：在使用者按下執行按鈕前，使用 `shlex.split(params)` 進行預檢。若拋出 `ValueError: No closing quotation`（引號未成對閉合），立即以紅框標示錯誤欄位並跳出提示阻擋執行。
- **路徑特徵自動修復 (Path Normalization)**：
  - **技術明訂**：針對包含 `\` 或 `/` 且看似路徑的參數，程式會自動以 `os.path.normpath()` 統一斜線格式。並使用 `strip()` 清除前後因複製貼上誤帶入的空白。若使用者輸入如 `%APPDATA%`，則以 `os.path.expandvars()` 將其展開為實體路徑。
- **進階輸入區語法突顯 (Syntax Highlighting)**：
  - **技術明訂**：在 `...` 的 `Text` 編輯區域內，套用基礎正則表達式 (Regex) 來高亮標記：以 `--` 或 `-` 開頭的 Flag (藍色)、被 `""` 或 `''` 包圍的字串 (綠色)。讓使用者能憑肉眼立刻檢查是否漏打引號。
- **檔案存在與副檔名檢查**：
  - **技術明訂**：執行前以 `os.path.exists()` 檢查檔案是否還在，若消失則 UI 反灰或跳出錯誤。若副檔名非預期的 `.py, .exe, .bat, .cmd, .ps1`，給予警告但不強制阻擋。
- **執行狀態捕捉與安全中斷**：
  - **技術明訂**：若腳本執行失敗 (`returncode != 0`)，自動捕捉並顯示 `stderr` 錯誤日誌。且 UI 必須提供「強制停止」按鈕（綁定 `process.terminate()` 或 `process.kill()`），防止腳本無窮迴圈或卡死。
- **終端機輸出與編碼防呆**：
  - **技術明訂**：底層執行必須設定 `errors='replace'` (例如於 `subprocess.Popen`)，防止遇到 Big5/CP950 編碼或特殊字元無法解析而導致主程式崩潰。並針對 Windows 環境加入 `creationflags=subprocess.CREATE_NO_WINDOW` 參數，確保背景執行時不會有黑色終端機視窗彈出干擾。

## 3. 使用者操作流程 (UX Flow)
1. **啟動程式**：自動從同目錄下的設定檔讀取並還原先前的頁籤、腳本與參數狀態。
2. **設定腳本**：在特定頁籤中新增腳本 -> 透過檔案選擇器載入 `run_task.bat` -> 填寫前 3 個常用參數欄位 -> 如有需要，點擊 `...` 貼上進階設定字串。
3. **拖拉整理**：若腳本過多，直接拖拉腳本項目調整上下順序；拖拉頁籤標籤調整左右順序。
4. **執行與檢視**：點擊腳本旁的執行鍵 -> 介面立即顯示即將執行的完整 CLI 指令 -> 腳本啟動。
5. **關閉程式**：自動將最新狀態覆寫回原設定檔中。
